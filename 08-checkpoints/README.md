# Checkpoints（检查点）与 Rewind（回退）

Checkpoints 允许你保存对话状态并回退到 Claude Code 会话中的之前节点。这对于探索不同的实施方案、从错误中恢复或比较替代方案非常有价值。

## 概览

Checkpoints 允许你保存对话状态并回退到之前的点，实现安全地实验和探索多种方案。它是对话状态的快照，包括：
- 所有交换的消息
- 所做的文件修改
- 工具使用历史
- 会话上下文

## 核心概念

| 概念 | 说明 |
|------|------|
| **Checkpoint（检查点）** | 包含消息、文件和上下文的对话状态快照 |
| **Rewind（回退）** | 返回到之前的检查点，丢弃后续变更 |
| **Branch Point（分支点）** | 从该检查点开始探索多种方案的分支节点 |

## 访问 Checkpoints

你可以通过两种主要方式访问和管理检查点：

### 使用键盘快捷键
按两次 `Esc` 键（`Esc` + `Esc`）打开检查点界面并浏览已保存的检查点。

### 使用斜杠命令
使用 `/rewind` 命令（别名：`/checkpoint`）快速访问：

```bash
# 打开回退界面
/rewind

# 或使用别名
/checkpoint
```

## 回退选项

当你执行回退操作时，会看到一个包含五个选项的菜单：

1. **恢复代码和对话** — 将文件和消息都还原到该检查点的状态
2. **仅恢复对话** — 只回退消息，保持当前代码不变
3. **仅恢复代码** — 只还原文件变更，保留完整的对话历史
4. **从此处总结** — 从该点开始将后续对话压缩为 AI 生成的摘要而非丢弃。原始消息保留在记录中。你可以选择性地提供指令来聚焦特定主题的摘要。
5. **算了** — 取消并返回当前状态

## 自动检查点

Claude Code 会自动为你创建检查点：

- **每次用户输入** — 每次用户发送消息时会创建新检查点
- **持久化存储** — 检查点跨会话持久保存
- **自动清理** — 检查点会在 30 天后被自动清理

这意味着你始终可以回退到对话中的任何之前节点，无论是几分钟前还是几天前的。

## 使用场景

| 场景 | 工作流程 |
|------|----------|
| **探索方案** | 保存 → 尝试方案 A → 保存 → 回退 → 尝试方案 B → 对比 |
| **安全重构** | 保存 → 重构 → 测试 → 失败则回退 |
| **A/B 测试** | 保存 → 设计 A → 保存 → 回退 → 设计 B → 对比 |
| **错误恢复** | 发现问题 → 回退到最后一个正常状态 |

## 实际示例

### 示例 1：探索不同方案

```
用户: 我们给 API 添加一个缓存层吧

Claude: 我来给 API 端点添加 Redis 缓存...
[在检查点 A 处做了修改]

用户: 实际上，还是试试内存缓存吧

Claude: 我来回退以探索另一种方案...
[用户按 Esc+Esc 并回退到检查点 A]
[在检查点 B 处实现了内存缓存]

用户: 现在我可以对比两种方案了
```

### 示例 2：从错误中恢复

```
用户: 把认证模块重构为使用 JWT

Claude: 我来重构认证模块...
[做了大量修改]

用户: 等等，这样破坏了 OAuth 集成。我们回去吧。

Claude: 我来帮你回退到重构之前的状态...
[用户按 Esc+Esc 并选择重构前的检查点]

用户: 这次我们用更保守的方式
```

### 示例 3：安全实验

```
用户: 我们试着用函数式风格重写这个
[实验前创建了检查点]

Claude: [做了实验性修改]

用户: 测试失败了。回退吧。
[用户按 Esc+Esc 并回退到检查点]

Claude: 已回退变更。让我们换一种方法试试。
```

## 最佳实践

由于检查点是自动创建的，你可以专注于工作而无需担心手动保存状态。但请记住以下实践：

### 有效使用检查点

✅ **推荐做法**：
- 回退前先审阅可用检查点
- 当你想探索不同方向时使用回退功能
- 保留检查点以便对比不同方案
- 理解每种回退选项的作用（恢复代码+对话、仅恢复对话、仅恢复代码、或摘要）

❌ **避免事项**：
- 不要仅依赖检查点来保护代码
- 不要期望检查点跟踪外部文件系统的变更
- 不要把检查点当作 git commit 的替代品

## 配置

你可以在设置中切换自动检查点：

```json
{
  "autoCheckpoint": true
}
```

- `autoCheckpoint`：是否在每次用户输入时自动创建检查点（默认：`true`）

## 局限性

检查点有以下限制：

- **Bash 命令变更不跟踪** — 文件系统上的 `rm`、`mv`、`cp` 等操作不会被捕获到检查点中
- **外部变更不跟踪** — 在 Claude Code 外部（如编辑器、终端等）所做的变更不会被捕获
- **不能替代版本控制** — 请使用 git 进行永久性的、可审计的代码库变更

## 与 Git 配合使用

Checkpoints 与 Git 互补（但不替代 Git）：

| 特性 | Git | Checkpoints |
|---------|-----|-------------|
| 作用范围 | 文件系统 | 对话 + 文件 |
| 持久性 | 永久 | 基于会话 |
| 粒度 | 提交 | 任意时间点 |
| 速度 | 较慢 | 即时 |
| 分享 | 是 | 有限 |

配合使用建议：
1. 用检查点进行快速实验
2. 用 git commit 保存最终确定的变更
3. 在 git 操作前创建检查点
4. 将成功的检查点状态提交到 git

> 💡 **中文开发者提示**：检查点是 Claude Code 最实用也最容易被忽视的功能之一。养成"大改动前先确认有检查点"的习惯，可以让你放心大胆地尝试各种重构方案。按 `Esc` 两次是最快的访问方式。
