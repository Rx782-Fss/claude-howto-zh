# Checkpoints 与 Rewind（回退）

Checkpoints 允许你保存对话状态并回退到 Claude Code 会话中的之前节点。这对于探索不同方案、从错误中恢复或比较替代方案非常有价值。

## 概览

Checkpoints 允许你保存对话状态并回退到之前的点，实现安全地实验和探索多种方案。它们是对话状态的快照，包括：
- 所有交换的消息
- 所做的文件修改
- 工具使用历史
- 会话上下文

Checkpoints 在探索不同方案、从错误中恢复或比较替代方案时非常有价值。

## 核心概念

| 概念 | 说明 |
|------|------|
| **Checkpoint** | 包含消息、文件和上下文的对话状态快照 |
| **Rewind** | 返回到之前的检查点，丢弃后续变更 |
| **Branch Point** | 从该检查点开始探索多种方案的分支节点 |

## 访问 Checkpoints

你可以通过两种主要方式访问和管理检查点：

### 使用键盘快捷键

按两次 `Esc` 键（`Esc` + `Esc`）打开检查点界面并浏览已保存的检查点。

### 使用斜杠命令

使用 `/rewind` 命令（别名：`/checkpoint`）快速访问：

```bash
# 打开回退界面
/rewind

# 或使用别名
/checkpoint
```

## 回退选项

当你执行回退操作时，会看到一个包含五个选项的菜单：

① **恢复代码和对话**——将文件和消息都还原到该检查点的状态
② **仅恢复对话**——只回退消息，保持当前代码不变
③ **仅恢复代码**——只还原文件变更，保留完整的对话历史
④ **从此处总结**——从该点开始将后续对话压缩为 AI 生成的摘要，释放上下文窗口空间。所选点之前的消息保持不变。磁盘上的文件不会被更改。原始消息保留在会话记录中。你可以选择性地提供指令来聚焦特定主题的摘要。
⑤ **算了**——取消并返回当前状态

> **注意**：恢复对话或总结后，所选消息的原始提示词会恢复到输入框中，以便你重新发送或编辑。

## 自动检查点

Claude Code 会自动为你创建检查点：

- **每次用户输入**——每次用户发送消息时会创建新检查点
- **持久化存储**——检查点跨会话持久保存
- **自动清理**——检查点会在 30 天后被自动清理

这意味着你始终可以回退到对话中的任何之前节点，无论是几分钟前还是几天前的。

## 使用场景

| 场景 | 工作流程 |
|------|----------|
| **探索方案** | 保存 → 尝试 A → 保存 → 回退 → 尝试 B → 对比 |
| **安全重构** | 保存 → 重构 → 测试 → 失败则回退 |
| **A/B 测试** | 保存 → 设计 A → 保存 → 回退 → 设计 B → 对比 |
| **错误恢复** | 发现问题 → 回退到最后一个正常状态 |

## 使用 Checkpoints

### 查看与回退

按两次 `Esc` 或使用 `/rewind` 打开检查点浏览器。你会看到所有可用检查点的列表及时间戳。选择任意检查点即可回退到该状态。

### 检查点详情

每个检查点显示：
- 创建时间戳
- 被修改的文件
- 对话中的消息数量
- 已使用的工具

## 实战示例

### 示例 ①：探索不同方案

```
用户: 我们给 API 添加一个缓存层吧

Claude: 我来给 API 端点添加 Redis 缓存...
[在检查点 A 处做了修改]

用户: 实际上，还是试试内存缓存吧

Claude: 我来回退以探索另一种方案...
[用户按 Esc+Esc 并回退到检查点 A]
[在检查点 B 处实现了内存缓存]

用户: 现在我可以对比两种方案了
```

### 示例 ②：从错误中恢复

```
用户: 把认证模块重构为使用 JWT

Claude: 我来重构认证模块...
[做了大量修改]

用户: 等等，这样破坏了 OAuth 集成。我们回去吧。

Claude: 我来帮你回退到重构之前的状态...
[用户按 Esc+Esc 并选择重构前的检查点]

用户: 这次我们用更保守的方式
```

### 示例 ③：安全实验

```
用户: 我们试着用函数式风格重写这个
[实验前创建了检查点]

Claude: [做了实验性修改]

用户: 测试失败了。回退吧。
[用户按 Esc+Esc 并回退到检查点]

Claude: 已回退变更。让我们换一种方法试试。
```

### 示例 ④：分支对比

```
用户: 我想对比两种数据库设计方案
[记录下检查点 —— 称之为 "起点"]

Claude: 我来创建第一个方案...
[实现方案 A]

用户: 现在回到起点尝试第二种方案
[用户按 Esc+Esc 并回退到 "起点"]

Claude: 现在我来实现方案 B...
[实现方案 B]

用户: 太好了！现在我有两个方案可以选择了
```

## 检查点保留策略

Claude Code 自动管理你的检查点：

- 每次用户输入自动创建检查点
- 旧检查点最多保留 30 天
- 检查点会自动清理以防止无限增长

## 工作流模式

### 探索分支策略

探索多种方案时：

```
① 初始实现 → 检查点 A
② 尝试方案 1 → 检查点 B
③ 回退到检查点 A
④ 尝试方案 2 → 检查点 C
⑤ 对比 B 和 C 的结果
⑥ 选择最佳方案并继续
```

### 安全重构模式

进行重大变更时：

```
① 当前状态 → 检查点（自动）
② 开始重构
③ 运行测试
④ 测试通过 → 继续工作
⑤ 测试失败 → 回退并尝试不同方案
```

## 最佳实践

由于检查点是自动创建的，你可以专注于工作而无需担心手动保存状态。但请记住以下实践：

### 有效使用 Checkpoints

**推荐做法：**
- 回退前先审阅可用检查点
- 当你想探索不同方向时使用回退功能
- 保留检查点以便对比不同方案
- 理解每种回退选项的作用（恢复代码+对话、仅恢复对话、仅恢复代码、或总结）

**避免事项：**
- 不要仅依赖检查点来保护代码
- 不要期望检查点跟踪外部文件系统的变更
- 不要把检查点当作 git commit 的替代品

## 配置

Checkpoints 是 Claude Code 的内置默认行为，无需任何配置即可启用。每次用户输入都会自动创建检查点。

唯一相关的设置是 `cleanupPeriodDays`，控制会话和检查点的保留时长：

```json
{
  "cleanupPeriodDays": 30
}
```

- `cleanupPeriodDays`：保留会话历史和检查点的天数（默认：`30`）

## 局限性

Checkpoints 有以下限制：

- **Bash 命令变更不跟踪**——文件系统上的 `rm`、`mv`、`cp` 等操作不会被捕获到检查点中
- **外部变更不跟踪**——在 Claude Code 外部（如编辑器、终端等）所做的变更不会被捕获
- **不能替代版本控制**——请使用 git 进行永久性的、可审计的代码库变更

## 故障排查

### 检查点丢失

**问题**：预期的检查点未找到

**解决方案**：
- 检查检查点是否被清除
- 检查磁盘空间
- 确保 `cleanupPeriodDays` 设置得足够高（默认：30 天）

### 回退失败

**问题**：无法回退到检查点

**解决方案**：
- 确保没有未提交的更改冲突
- 检查检查点是否损坏
- 尝试回退到其他检查点

## 与 Git 配合使用

Checkpoints 与 Git 互补（但不替代 Git）：

| 特性 | Git | Checkpoints |
|------|-----|-------------|
| 作用范围 | 文件系统 | 对话 + 文件 |
| 持久性 | 永久 | 基于会话 |
| 粒度 | 提交 | 任意时间点 |
| 速度 | 较慢 | 即时 |
| 分享 | 是 | 有限 |

配合使用建议：
1. 用检查点进行快速实验
2. 用 git commit 保存最终确定的变更
3. 在 git 操作前创建检查点
4. 将成功的检查点状态提交到 git

## 快速入门指南

### 基本工作流

① **正常工作**——Claude Code 自动创建检查点
② **想返回？**——按两次 `Esc` 或使用 `/rewind`
③ **选择检查点**——从列表中选择要回退的点
④ **选择恢复内容**——选择恢复代码+对话、仅恢复对话、仅恢复代码、从此处总结或取消
⑤ **继续工作**——你已经回到那个点了

### 键盘快捷键

- **`Esc` + `Esc`**——打开检查点浏览器
- **`/rewind`**——访问检查点的另一种方式
- **`/checkpoint`**——`/rewind` 的别名

## 何时应该回退：上下文监控

Checkpoints 让你能回退——但你如何知道**何时**应该回退？随着对话增长，Claude 的上下文窗口会被填满，模型质量会悄然下降。你可能在一个半盲状态下推送代码而浑然不觉。

**[cc-context-stats](https://github.com/luongnv89/cc-context-stats)** 通过在 Claude Code 状态栏添加实时**上下文区域**来解决这个问题。它追踪你在上下文窗口中的位置——从 **Plan**（绿色，安全地进行规划和编码）经过 **Code**（黄色，避免开始新的规划）到 **Dump**（橙色，结束并回退）。当你看到区域切换时，就知道是时候创建检查点重新开始了，而不是在输出质量下降的情况下硬撑。

## 总结

Checkpoints 是 Claude Code 的自动功能，让你能安全地探索不同方案而不必担心丢失工作。每次用户输入都会自动创建新检查点，因此你可以回退到会话中的任意之前节点。

核心优势：
- 无畏地尝试多种方案
- 快速从错误中恢复
- 并排比较不同解决方案
- 安全地与版本控制系统集成

记住：Checkpoints 不是 git 的替代品。用 Checkpoints 进行快速实验，用 git 保存永久性的代码变更。
