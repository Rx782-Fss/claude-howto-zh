# AI 代码生成的整洁代码规则

这些规则指导代码生成,使其产出可维护的、专业质量的代码。

## 有意义的命名
- 使用能揭示意图的名称,解释某事物为何存在
- 避免误导性名称和无意义的区分(如 `data`、`info`、`manager`)
- 使用可发音、可搜索的名称
- 类名用名词(如 `UserAccount`、`PaymentProcessor`)
- 方法名用动词(如 `calculateTotal`、`sendEmail`)
- 避免心智映射和编码(匈牙利命名法、前缀)

## 函数
- 保持函数精简(理想情况下少于 20 行)
- 只做一件事 ---- 单一职责原则(Single Responsibility Principle)
- 每个函数只有一个抽象层次
- 限制参数数量:理想 0-2 个,最多 3 个,避免标志参数
- 无副作用 ---- 函数应做其名称所表达的事
- 将命令(改变状态)与查询(返回信息)分离
- 优先使用异常而非错误码

## 注释
- 代码应自解释 ---- 尽可能避免注释
- 好的注释:法律信息、警告、TODO、公共 API 文档
- 差的注释:冗余的、误导性的、为烂代码做解释的注释
- 永远不要注释掉代码 ---- 直接删除它(版本控制会保留历史)
- 如果需要注释,考虑重构代码来替代

## 格式
- 保持文件小巧且聚焦
- 垂直格式:相关概念放在一起,空行分隔概念
- 水平格式:限制行长(80-120 字符)
- 使用一致的缩进和团队风格
- 将相关的函数分组在一起

## 对象和数据结构
- 对象:将数据隐藏在抽象后面,通过方法暴露行为
- 数据结构:暴露数据,行为最少化
- 得墨忒尔法则(Law of Demeter):只与直接朋友对话,避免 `a.getB().getC().doSomething()`
- 不要盲目地通过 getter/setter 暴露内部结构

## 错误处理
- 使用异常,而非返回码或错误标志
- 当代码可能失败时,先写 `try-catch-finally`
- 在异常消息中提供上下文
- 不要返回 `null` ---- 返回空集合或使用 Optional/Maybe
- 不要传递 `null` 作为参数

## 类
- 小类:按职责衡量,而非按行数
- 单一职责原则:只有一个修改理由
- 高内聚:类变量被多个方法使用
- 低耦合:类之间的依赖最少化
- 开闭原则(Open/Closed Principle):对扩展开放,对修改关闭

## 单元测试
- 快速(Fast)、独立(Independent)、可重复(Repeatable)、自验证(Self-validating)、及时(Timely)---- F.I.R.S.T. 原则
- 每个测试一个断言(或一个概念)
- 测试代码质量应等同于生产代码质量
- 可读的测试名称,描述被测内容
- Arrange-Act-Assert(准备-执行-断言)模式

## 代码质量原则
- **DRY(Don't Repeat Yourself,不要重复自己)**:杜绝重复
- **YAGNI(You Aren't Gonna Need It,你并不需要它)**:不为假设的未来构建功能
- **KISS(Keep It Simple,保持简单)**:避免不必要的复杂性
- **童子军规则**:离开时的代码比你到来时更整洁

## 应避免的代码异味
- 过长的函数或类
- 重复代码
- 死代码(未使用的变量、函数、参数)
- 特性依恋(方法对其他类的兴趣超过自身类)
- 不当亲密(类之间了解过多)
- 过长参数列表
- 基本类型偏执(过度使用基本类型而非小对象)
- Switch/case 语句(考虑多态替代)
- 临时字段(类变量仅在某些时候使用)

## 并发
- 将并发代码与其他代码分离
- 限制同步/锁定数据的范围
- 使用线程安全的集合
- 保持同步段短小
- 了解你的执行模型和原语

## 系统设计
- 构造与使用分离(依赖注入)
- 为复杂对象创建使用工厂、建造者模式
- 面向接口编程,而非面向实现
- 优先使用组合而非继承
- 仅在简化设计时应用设计模式,不要为了炫技而用

## 重构
- 持续重构,而不是大批量一次性重构
- 前后都要有通过的测试
- 小步前进:一次只改一处
- 常见重构手法:提取方法(Extract Method)、重命名(Rename)、搬移(Move)、内联(Inline)

## 文档
- 自文档化的代码 > 注释 > 外部文档
- 公共 API 需要清晰的文档
- 文档中包含示例
- 让文档靠近代码(最好就在代码中)

---

**核心理念**:代码被阅读的次数是编写的 10 倍以上。优化可读性和可维护性,而不是追求聪明技巧。
