## 报告漏洞

我们重视社区的安全研究人员在维护 Claude How To 项目安全方面所做的努力。如果你发现了安全漏洞，请**不要**通过公开 Issue 或 PR 来报告。

### 如何报告

请发送邮件至：[请在仓库中设置安全邮箱]

你的报告应包含：

- **漏洞描述**：问题的清晰且可复现的描述
- **影响范围**：谁可能受到此问题的影响
- **复现步骤**：详细的重现步骤
- **建议修复方案**（可选）：建议的修复方案
- **联系方式**：我们可以联系到你的方式

### 你可以期望什么

- **24 小时内确认**收到你的报告
- **48 小时内初步评估**
- **7 天内更新进度**（或解释延迟原因）
- **修复发布前保密处理**

### 我们承诺

- **不追究责任**：以善意的方式开展安全研究
- **及时响应**：及时回应所有安全报告
- **协调披露**：与研究人员协调修复和公开披露时间表
- **致谢**：在安全公告中认可贡献者（除非你希望保持匿名）

## 安全范围

### 范围内

以下内容属于本安全策略的范围：
- 本仓库中的代码执行漏洞
- 社区工程中的依赖项漏洞
- 认证和授权缺陷
- 注入攻击（XSS、命令注入等）
- 敏感信息泄露
- 配置安全问题

### 范围外

以下内容不属于范围：
- 需要物理访问系统的问题
- 需要已有管理员权限的问题
- 第三方服务或依赖项中的问题（请直接向其报告）
- 理论上可行但实际无法利用的问题
- 最佳实践改进（欢迎通过普通 Issue 提出）
- DoS 攻击（标准速率限制缓解措施除外）

## 安全最佳实践

### 对于用户

1. **审查代码**：安装前始终审查来自本仓库的代码
2. **使用最新版本**：保持更新至包含安全补丁的最新版本
3. **环境变量**：永远不要将 API 密钥或 Token 硬编码到文件中
4. **权限**：遵循最小权限原则
5. **验证来源**：从官方 GitHub 仓库下载

### 对于贡献者

1. **输入校验**：对所有用户输入进行校验和清洗
2. **依赖管理**：定期更新依赖项并检查已知漏洞
3. **敏感数据**：永远不要提交密钥或凭证
4. **错误消息**：不要在错误消息中泄露敏感信息
5. **代码审查**：对变更进行安全审查

### 对于钩子和脚本

1. **权限控制**：脚本应具有最小必要权限
2. **输入验证**：验证所有外部输入
3. **错误处理**：优雅地处理错误而不泄露信息
4. **日志记录**：不要记录敏感数据
5. **沙箱化**：尽可能在受限环境中运行

## 已知安全注意事项

### API 密钥和 Token
示例文件可能包含占位符值。**切勿在生产环境中使用这些值**。

```bash
# 始终使用环境变量
export GITHUB_TOKEN="your_actual_token"
export DATABASE_URL="postgresql://..."
```

### MCP 服务器
MCP 服务器在你的本地环境中运行。确保你信任正在使用的服务器实现。

### 钩子执行
钩子在你请求的操作前后自动运行。审查任何自定义钩子，以确保其行为符合预期。

### 插件
插件可能包含第三方代码。在安装前审查插件源码，特别是那些来自非 Anthropic 官方市场的插件。

## 安全更新流程

当发现安全问题时：

1. **确认并评估**（0-2 天）
2. **开发修复方案**（1-7 天，视严重程度而定）
3. **测试修复方案**（1-3 天）
4. **准备安全公告**（与修复同步进行）
5. **发布修复**（协调后立即发布）
6. **公开披露**（修复可用后 7-14 天，或按约定）

### 严重程度级别

| 级别 | 描述 | 响应时间 |
|------|------|----------|
| **Critical（严重）** | 可被轻易利用，影响大量用户 | 24-48 小时 |
| **High（高）** | 可被利用，有显著影响 | 7 天 |
| **Medium（中）** | 较难利用，影响有限 | 30 天 |
| **Low（低）** | 影响小，难以利用 | 下一个版本 |
